# Конфигуратор ТЗ по КТРУ

Веб-приложение для подготовки технического задания по позициям КТРУ. Пользователь выбирает группу товара, уточняющий признак, обязательные и дополнительные характеристики, добавляет позиции в черновик ТЗ, проверяет предпросмотр и выгружает результат в DOCX, XLSX или PDF.

Проект сделан как MVP для локального запуска и демонстрации полного сценария: регистрация пользователя, одобрение администратором, настройка КТРУ, сборка ТЗ и экспорт документов.

## Возможности

- Регистрация пользователей с одобрением через Django Admin.
- Роли `admin` и `user`.
- Поиск и выбор КТРУ по 6 группам: системный блок, монитор, клавиатура, мышь, моноблок, ноутбук.
- Выпадающий список уточняющих признаков для выбранной группы.
- Автоматическая сортировка уточняющих значений по числовому или алфавитному порядку.
- Конфигуратор характеристик:
  - обязательные и необязательные характеристики;
  - одиночный и множественный выбор;
  - исключение необязательных характеристик;
  - фильтры по обязательным, необязательным, выбранным и характеристикам без единицы измерения.
- Черновики ТЗ:
  - создание;
  - копирование;
  - удаление;
  - добавление нескольких позиций;
  - редактирование уже добавленных позиций;
  - предпросмотр.
- Приписки под таблицей:
  - шаблоны приписок из админки;
  - ручные приписки в черновике;
  - вывод приписок в предпросмотре и DOCX.
- Экспорт:
  - DOCX через `python-docx`;
  - XLSX через `openpyxl`;
  - PDF через LibreOffice headless из DOCX.

## Технологии

- Backend: Django, Django REST Framework, PostgreSQL, Django Admin.
- Frontend: React, Vite, TypeScript, Tailwind CSS.
- Runtime: Docker Compose, Nginx, Gunicorn, PostgreSQL.
- Документы: `python-docx`, `openpyxl`, LibreOffice headless.


## Быстрый запуск

1. Создайте файл окружения:

```powershell
Copy-Item .env.example .env
```

2. Соберите и запустите контейнеры:

```powershell
docker compose up --build
```

3. В другом терминале примените миграции:

```powershell
docker compose exec backend python manage.py migrate
```

4. Создайте администратора:

```powershell
docker compose exec backend python manage.py createsuperuser
```

5. Загрузите тестовые данные КТРУ:

```powershell
docker compose exec backend python manage.py seed_ktru
```

После запуска:

- пользовательский интерфейс: `http://YOUR_SERVER_IP:8081/`
- админка: `http://YOUR_SERVER_IP:8081/admin/`

## Если на сервере уже крутится другой Docker-проект

Этот проект по умолчанию публикует nginx на `8081`, чтобы не конфликтовать с другим приложением на том же VPS.

Если тебе нужен другой внешний порт, меняется только проброс в `docker-compose.yml` и два URL в `.env`:

- `DJANGO_CSRF_TRUSTED_ORIGINS`
- `FRONTEND_ORIGIN`

## Руководство пользователя

### 1. Регистрация и вход

1. Откройте `http://YOUR_SERVER_IP:8081/`.
2. На вкладке регистрации укажите email и пароль.
3. После регистрации аккаунт попадет в ожидание одобрения.
4. Администратор должен открыть Django Admin и одобрить заявку.
5. После одобрения пользователь может войти в приложение.

### 2. Создание ТЗ

1. В рабочем кабинете введите название нового ТЗ.
2. Нажмите `Создать`.
3. Откроется экран редактирования черновика.

### 3. Добавление позиции

1. Выберите группу КТРУ.
2. Выберите уточняющий признак из выпадающего списка.
3. Укажите количество.
4. Нажмите `Определить`.
5. Выберите значения обязательных характеристик.
6. При необходимости выберите необязательные характеристики.
7. Ненужные необязательные характеристики можно выключить кнопкой с минусом.
8. Нажмите `Добавить`.

Позиция появится в списке слева.

### 4. Редактирование позиции

1. Откройте черновик.
2. В левой панели выберите нужную позицию или нажмите `Редактировать`.
3. Форма заполнится текущими данными позиции.
4. Измените группу, уточняющий признак, количество или характеристики.
5. Нажмите `Сохранить`.

### 5. Удаление позиции

В левой панели у позиции нажмите кнопку удаления. Позиция будет удалена из черновика.

### 6. Приписки

В редакторе есть блок `Приписки`.

Можно:

- выбрать один или несколько шаблонов;
- добавить собственный текст;
- указать несколько ручных приписок, каждая строка станет отдельным пунктом.

После настройки нажмите `Сохранить`.

Приписки выводятся:

- в предпросмотре;
- в DOCX под таблицей характеристик каждой позиции.

### 7. Предпросмотр и экспорт

1. После добавления позиций нажмите `Предпросмотр`.
2. Проверьте таблицы и приписки.
3. Нажмите `Готово`, если ТЗ готово.
4. Скачайте нужный формат:
   - `DOCX`;
   - `XLSX`;
   - `PDF`.

## Руководство администратора

### Одобрение пользователей

1. Откройте `http://YOUR_SERVER_IP:8081/admin/`.
2. Перейдите в `Registration requests`.
3. Выберите заявку.
4. Используйте действие одобрения или измените статус вручную.

До одобрения пользователь не сможет войти в рабочий интерфейс.

### Управление КТРУ

В админке доступны:

- группы КТРУ;
- позиции КТРУ;
- характеристики;
- значения характеристик;
- связи укрупненных и уточненных позиций;
- журналы загрузки.

Для повторной загрузки fixture:

```powershell
docker compose exec backend python manage.py seed_ktru
```

Для загрузки с Excel:

```powershell
docker compose exec backend python manage.py seed_ktru --excel "/app/fixtures/Краткое КТРУ.xlsx"
```

### Шаблоны приписок

1. Откройте Django Admin.
2. Перейдите в `Postscript templates`.
3. Создайте шаблон:
   - `name` - короткое название;
   - `text` - текст приписки;
   - `is_active` - включить отображение в интерфейсе.
4. Активные шаблоны будут доступны пользователю в редакторе черновика.

## Экспорт DOCX

DOCX формируется в табличном виде:

- верхняя таблица позиции;
- строка требований заказчика;
- таблица характеристик;
- приписки под таблицей.

По умолчанию используется:

- шрифт Times New Roman;
- размер 10 pt;
- авто-высота строк по содержимому;
- компактные отступы внутри ячеек;
- альбомная ориентация при наличии позиций.

Для характеристик без единицы измерения инструкция меняется на:

```text
Значение характеристики не может изменяться участником закупки
```

## Важное предупреждение по паролям

По бизнес-требованию MVP содержит отдельное зашифрованное поле `encrypted_password_for_admin`, чтобы администратор мог посмотреть пароль пользователя в админке.

Основной пароль для входа хранится штатно и безопасно как Django password hash. Но сама возможность просмотра исходного пароля администратором является компромиссом и не рекомендуется для промышленной эксплуатации. Для production эту функцию нужно удалить и заменить стандартным сбросом пароля.

## Что не входит в MVP

- Редактор Word-шаблонов через UI.
- Пользовательский импорт КТРУ через UI.
- Полноценная интеграция с live API ЕИС.
- Многостраничный workflow согласования ТЗ.
- Production-настройки безопасности, мониторинга и резервного копирования.
